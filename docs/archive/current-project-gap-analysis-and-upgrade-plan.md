# 当前项目缺口分析与升级方案

## 1. 先给结论

当前项目已经不是空壳，它已经具备了一个真实可运行的底座：

- 能初始化 A 股上市公司全集
- 能从巨潮接口同步一部分公司的当前高管与董事会快照
- 能落库公司、人物、当前快照、任职区间、同步运行记录
- 能展示首页、公司页、人物页

但它**还不是一个完整的“中国版上市公司高管变动系统”**。  
现在最准确的定义是：

**它是一个“当前高管基线库原型”，而不是“完整的人事事件情报产品”。**

所以升级不能乱补功能，必须按顶层结构往前长。正确做法是：

1. 先把基线库产品化，让覆盖、检索、数据质量、运维可见
2. 再把公告事件流接进来
3. 再做指标、预警、人物流动和董事会重叠图

这次升级里，我已经先把第 1 步的骨架补上了：

- 覆盖台账页
- 公司库检索页
- 人物库检索页
- 对应 API
- 覆盖率、交易所覆盖、板块覆盖、核心角色覆盖、待同步队列、最近同步任务

这样项目才从“能跑”变成“能管理、能观察、能扩展”。

## 2. 当前项目的真实状态

### 2.1 已经实现的模块

#### A. 数据底座

- `Company`：上市公司主表
- `Person`：人物主表
- `ExecutiveSnapshot`：当前高管快照
- `RoleTenure`：当前推导出的任职区间
- `BaselineRun`：同步任务运行记录

#### B. 基础采集

- 证券清单初始化
- 公司简介抓取
- 公司高管抓取
- 角色归一
- 当前快照入库

#### C. 基础产品层

- 首页概览
- 公司详情页
- 人物详情页
- 基础 JSON API

### 2.2 已经存在但基本没真正启用的模块

这些模块的表结构已经在代码里，但目前还没有形成真实业务闭环：

- `SourceDocument`
- `Event`
- `CompanyMetricDaily`

说明当前系统还没有真正走到：

- 公告抓取
- 公告分类
- 人事事件抽取
- 事件去重
- 指标回填

### 2.3 当前最核心的问题

不是“界面不够多”，而是这几个结构性缺口：

1. 缺少覆盖管理层  
   之前只能看少量页面，看不到整个库同步到了哪里、缺口在哪、哪些市场落后。

2. 缺少检索层  
   没有公司库与人物库检索，意味着数据存在，但很难作为产品使用。

3. 缺少事件层  
   当前只有“快照”，没有“变动事件”，所以还不能回答“谁在什么时候发生了什么变化”。

4. 缺少指标层  
   `30 天变动次数`、`稳定度评分`、`同比环比` 这些都还没有真实生成。

5. 缺少任务治理  
   只有 CLI，没有真正的同步策略、失败重试、优先级、覆盖率追踪和监控规则。

6. 缺少质量层  
   没有测试、没有数据校验规则面板、没有系统级错误视图。

## 3. 顶层设计：项目应该长成什么样

这个项目接下来要遵循的顶层结构，不是“继续往当前页面上加功能”，而是形成下面 6 层。

### 第一层：市场基础层

职责：

- 上市公司全集
- 公司元数据
- 市场、板块、行业归类
- 是否活跃、是否退市

当前状态：

- 已有基础能力
- 还缺持续更新机制

### 第二层：当前基线层

职责：

- 当前在任人物快照
- 当前核心角色覆盖
- 当前任职区间
- 人物与公司关联

当前状态：

- 已落地
- 本次升级已补产品化检索与覆盖可视化

### 第三层：公告事件层

职责：

- 抓公告
- 判定是否为人事相关
- 抽取标准事件
- 去重与归一

这是项目从“库”变成“情报产品”的关键层。

当前状态：

- 数据结构预留了
- 业务链路尚未落地

### 第四层：指标与分析层

职责：

- 30/90 天变动次数
- 稳定度评分
- 同比/环比
- CFO/CEO/董事长异常变动预警

当前状态：

- 模型预留
- 没有真实计算任务

### 第五层：图谱与研究层

职责：

- 人物跨公司流动轨迹
- 董事会共享成员网络
- 独董重叠关系
- 产业/资本圈层连接

当前状态：

- 完全未开始

### 第六层：产品与运维层

职责：

- 检索
- 台账
- 订阅提醒
- 审核后台
- 任务监控
- 数据质量控制

当前状态：

- 这次升级补了一部分产品层
- 运维与审核后台仍缺

## 5. 本次升级的设计取舍

这次没有直接去做事件流，而是先做了“基线产品化”。这是故意的。

原因：

1. 当前项目最成熟的数据只有“公司全集 + 当前高管快照”
2. 如果连覆盖台账和检索层都没有，后面事件层做出来也很难管理
3. 产品可用性太弱，无法判断底层数据到底覆盖到什么程度
4. 先把基线层做稳，后续事件流才能落到一个清晰结构上

所以这次升级的目标被锁定为：

**把当前原型升级成“可管理的当前高管基线产品”。**

## 6. 本次已经做出来的升级

### 6.1 覆盖台账页

新增页面：

- `/coverage`

新增能力：

- 总体覆盖率
- 状态分布
- 交易所覆盖
- 板块覆盖
- 核心角色覆盖
- 最近同步任务
- 待同步公司队列

意义：

### 6.2 公司库检索页

新增页面：

- `/companies`

新增能力：

- 按关键词检索公司
- 按交易所筛选
- 按同步状态筛选
- 查看每家公司当前快照条数与核心角色条数

意义：

- 当前基线库终于能被系统化浏览，而不是只能跳点式打开公司页

### 6.3 人物库检索页

新增页面：

- `/people`

新增能力：

- 按人物姓名检索
- 按角色筛选
- 只看当前活跃或包含历史任职
- 展示当前任职摘要

意义：

- 人物库从“详情页存在”升级为“可检索的人物集合”

### 6.4 新 API

新增接口：

- `/api/coverage`
- `/api/companies`
- `/api/people`

这些接口让后续前端重构、独立前后端拆分或者外部调用更顺。

## 7. 后续完整升级路线

这部分是接下来真正该怎么做，不是零碎建议，而是正式路线。

### 阶段 A：把基线层彻底做稳

目标：

- 让当前高管基线覆盖全 A 股
- 完成覆盖率、检索、任务观察、数据质量基础

任务：

- 跑完整量同步
- 增加失败重试
- 增加最近同步时间与错误原因展示
- 增加人物去重辅助规则

验收标准：

- 活跃 A 股公司覆盖率达到高水平
- 页面可稳定浏览公司库与人物库
- 覆盖台账可清楚识别未同步队列

### 阶段 B：接入公告事件流

目标：

- 形成真正的“变动追踪”

任务：

- 抓取公告列表
- 存 `SourceDocument`
- 公告分类
- 抽取 `Event`
- 回填人物与任职区间

验收标准：

- 公司页出现真实事件时间线
- 人物页出现跨公司历史轨迹
- 首页最近事件流不再是空壳

### 阶段 C：补指标层

目标：

- 让产品具备研究价值

任务：

- 生成 `CompanyMetricDaily`
- 30/90 天变动
- 稳定度评分
- 环比/同比
- 异常离任提示

验收标准：

- 排行榜不再占位
- 指标有真实数值
- 能筛出治理波动公司

### 阶段 D：补图谱与订阅

目标：

- 形成差异化能力

任务：

- 人员流动路径
- 董事会重叠图
- 人物关系边
- 公司/人物/角色订阅提醒

验收标准：

- 能回答“这个人从哪里流向哪里”
- 能回答“这些公司董事会是否高度重叠”

## 8. 技术实现上的建议约束

为了避免项目后续越来越乱，这几个约束要固定住：

1. 不要把采集逻辑写进路由层
2. 不要让模板层自己拼业务逻辑
3. 事件抽取必须保留证据片段
4. 任何图谱能力都必须建立在标准事件之上
5. 人物消歧前期宁可保守，不要激进合并
6. 所有指标都必须可回溯到事件和快照来源

## 9. 现在这个项目的阶段判断

现在最准确的阶段判断是：

- 已经不是“想法”
- 也不是“完整产品”
- 而是一个已经搭起真实数据底座的 **V0.2 基线产品**

这次升级后，它的成熟度明显比之前高，因为它从：

- 只有首页/公司页/人物页的原型

升级成了：

- 有覆盖管理
- 有公司库
- 有人物库
- 有检索 API
- 有可观察运行记录

但它距离最终目标还差最关键的两层：

- 公告事件流
- 指标与图谱

## 10. 最后一句话

这个项目接下来不能再以“补一个页面、补一个接口”的方式推进。  
已经锁定的正确路径是：

**基线产品化 -> 事件流 -> 指标层 -> 图谱与订阅。**

这就是接下来整个项目的升级总纲。
